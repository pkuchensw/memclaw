默认：

- benchmark 运行在 **OpenClaw 真环境** 中，参考 WildClawBench 的组织方式：任务在隔离 workspace / Docker 中运行，任务模板包含 Prompt、Expected Behavior、Grading Criteria、Automated Checks、Workspace Path 等结构；WildClawBench 本身也是把 agent 放进 live OpenClaw 环境里跑真实端到端任务，并覆盖多模态、长流程、浏览器、bash、文件系统、邮件、日历等工具。([GitHub](https://github.com/InternLM/WildClawBench))
- 上下文压缩先默认使用 **lossless-claw / LCM** 作为插件式 context engine：它会把消息持久化到 SQLite，构建分层 DAG summary，每轮组装“摘要 + 近期原文”作为 active context，并提供 `lcm_grep`、`lcm_describe`、`lcm_expand` 这类回放工具；插件可以通过 `plugins.slots.contextEngine = "lossless-claw"` 接入。([GitHub](https://github.com/martian-engineering/lossless-claw))

# 一、项目目标

构建 **OpenClaw-MemBench**，围绕 8 个能力，每个能力设计 5 个场景，共 **40 个任务场景**。
每个任务都必须满足 4 个要求：

1. **可在 OpenClaw 中真实执行**
2. **天然包含工具调用**
3. **尽量覆盖多模态**
4. **预留上下文压缩评测位点**

这 40 个场景不是 40 个“单轮 prompt”，而是 40 个**多 episode 的 agent 任务**。每个任务都包含：

- 前置 episode：埋设约束 / 状态 / 历史经验 / 冲突证据 / 打断信息
- 执行 episode：agent 使用工具、文件、浏览器、代码等完成任务
- 压缩位点：在若干 turn / episode 后触发 LCM compaction
- 终局任务：检查压缩后是否还能完成任务

------

# 二、总设计原则

## 1. 任务不是 task-first，而是 capability-first

同一个“文献检索”可以测不同能力：

- 如果测刚给的格式和路径能否记住，那是 Recent Constraint Tracking
- 如果测后续更新的年份或输出格式能否覆盖旧要求，那是 Version Update
- 如果测上一个检索流程能否迁移到另一个站点，那是 Procedure Transfer

## 2. 每个任务都必须能在“全上下文”和“压缩上下文”下重复运行

统一跑 4 组 setting：

- `full-context`
- `lcm-medium-budget`
- `lcm-low-budget`
- `lcm-stress-budget`

核心指标不是只看成功率，而是看：

- `Retention = success_under_compression / success_under_full_context`

## 3. 每个任务都必须有 white-box oracle

不仅判最终结果，还要判：

- 记住了哪些约束
- 是否用了最新状态
- 是否调用了不该调用的旧经验
- 是否做了正确证据仲裁
- 是否把 episode 编译成了正确的 memory form

## 4. 默认把 LCM 当“上下文引擎”，不是 benchmark 的被测对象本身

第一版 benchmark 不强依赖 agent 必须显式调用 `lcm_grep` 等工具，但要：

- 记录是否发生了 compaction
- 记录 compaction 之后 agent 有没有显式 recall 行为
- 记录 recall 成败

# 三、能力—记忆对象映射

1. **Recent Constraint Tracking** → `context_cache`
2. **Version Update** → `state_memory`
3. **Procedure Transfer** → `procedural_memory`
4. **Repeated Mistake Prevention** → `anti_memory`
5. **Source Conflict Resolution** → `evidence_graph`
6. **Memory Operation Selection** → `memory_router`
7. **Goal Interruption / Task Resumption** → `context_cache + state_memory`

# 十五、LCM 接入规范

## 1. 默认配置

benchmark 默认把 LCM 作为 OpenClaw 的 `contextEngine`，因为 lossless-claw 就是这样接入的。([GitHub](https://github.com/Martian-Engineering/lossless-claw/blob/main/README_zh.md))

建议统一配置模板：

```json
{
  "plugins": {
    "slots": {
      "contextEngine": "lossless-claw"
    },
    "entries": {
      "lossless-claw": {
        "enabled": true,
        "config": {
          "contextThreshold": 0.75
        }
      }
    }
  }
}
```

## 2. 评测模式

每个任务跑 4 组：

- `baseline_full_context`
- `lcm_medium_budget`
- `lcm_low_budget`
- `lcm_stress_budget`

## 3. 记录项

必须记录：

- 是否发生 compaction
- compaction 前上下文 token
- compaction 后 active context token
- 是否调用 `lcm_grep` / `lcm_describe` / `lcm_expand`
- recall 后是否命中正确历史信息

LCM 本身保存 SQLite 数据并支持 DAG summary 与 recall 工具，所以非常适合作为第一版 benchmark 的默认压缩层。([GitHub](https://github.com/martian-engineering/lossless-claw))

# 十六、统一测试逻辑

这是给 code agent 最关键的一段。

## 每个任务的执行逻辑固定为 8 步

### Step 1. 初始化 workspace

- 拷贝任务文件
- 准备 PDF / 图像 / 视频 / repo / 邮件样本 / 日历样本
- 注入任务说明

### Step 2. 启动 OpenClaw agent

- 配置模型
- 配置工具权限
- 根据 run profile 决定是否启用 LCM 以及 budget 档位

### Step 3. 回放前置 episode

- 按 `scenario.jsonl` 逐轮喂给 agent
- episode 之间可插入工具输出、文件变更、外部更新

### Step 4. 触发压缩

- 按 `compression_checkpoints` 触发
- 记录压缩前后 token / context size
- 记录 recall 调用情况

### Step 5. 执行终局任务

- 给 agent 发最终指令
- 允许 agent 调工具完成任务

### Step 6. 收集轨迹

收集：

- agent 消息
- 工具调用日志
- 生成文件
- recall 记录
- compaction 记录

### Step 7. 运行 grader

grader 同时检查：

- 最终产物
- 中间行为
- 是否遵守能力逻辑

### Step 8. 聚合结果

输出：

- per-task metrics
- per-capability metrics
- budget-retention 曲线
- error taxonomy
