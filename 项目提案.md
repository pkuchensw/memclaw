# Openclaw-bench

capacity-based的benchmark + 可追溯、可更新、预算感知的上下文管理架构

Motivation：什么时候保留原文、什么时候抽取结构化记忆、什么时候做层级摘要、什么时候只保留指针并不是task-based，而应该做成Capability-based。根据agent的反馈进行知识更新、selective forgetting 与自适应压缩，从而提升模型在不同场景下能力的泛化性，并提升模型使用不能能力的针对性（防止使用错误的经验），同时提升系统efficiency。

核心motivation是如何衡量/提升agent系统在长上下文背景下的自我进化。按 agent 暴露出的能力缺陷来决定“写入哪种记忆、用哪种压缩形式、以及如何在未来回放

1. Context Meomry：存近期 raw context、关键槽位、未完成子目标。

2. State Memory：存事实、约束、用户偏好、环境状态、版本链。

3. Procedural Memory：存成功步骤模板、诊断顺序、修复方案、触发条件。

4. Anti-Memory：存错误模式、诱发条件、禁止动作、失效策略、快速检查清单。

5. Evidence Graph：存冲突证据、来源、时间、置信度、适用范围。

我们基于能力分类：

**Recent Constraint Tracking**：前面几轮刚给出格式、参数、路径或子目标约束，后面立即要求 agent 执行，看它能否保住当前任务所需的近期关键信息；**例如**用户刚要求“帮我搜索 2024–2025 的 arXiv 文献，只保留强化学习和 memory agent 方向，并输出成 CSV”，后面直接让 agent 开始检索与整理。

**Version Update**：前面某个事实、规则或配置被后续新信息覆盖，最后要求 agent 完成任务，看它是否使用最新版本而不是沿用旧状态；**例如**一开始要求用 Python 3.10 配环境，后面补充“改成 3.11 且必须用 uv，不要用 conda”，最后让 agent 生成并执行环境配置步骤。

**Procedure Transfer**：前面成功解决过一个小任务，后面来一个结构相似但表面不同的新任务，看 agent 能否复用先前抽象出的步骤模板而不是重新试错；**例如**前面已经成功完成一次“检索 arXiv → 筛标题摘要 → 去重 → 导出表格”的流程，后面换成“检索 ACL/ICLR 论文并整理到表格”，看是否复用同样 procedure。

**Repeated Mistake Prevention**：前面 agent 犯过某种错误并被纠正，后面给出相似情境，看它能否利用负反馈避免重复犯同类错误；**例如**前面配置环境时因为忘记先检查 CUDA 版本导致安装失败，后面在另一个项目里重新配环境时，看它会不会先检查驱动和 CUDA 兼容性。

**Source Conflict Resolution**：前面来自用户、工具、文件或环境状态的信息彼此冲突，后面要求 agent 决策或执行，看它能否选择正确证据来源；**例如**用户口头说项目依赖是 `torch==2.1`，但 `pyproject.toml` 和报错日志显示实际要求 `torch==2.3`，最后让 agent 修复环境并解释依据。

**Memory Operation Selection**：在不同 budget 和失败模式下给出长程任务，看 agent 能否自适应选择保留原文、写状态记忆、抽取 procedure、写 anti-rule 或只保留 pointer；**例如**在长程“调研 memory agent 论文并复现实验环境”的任务中，遇到文献筛选阶段应保留 procedure，遇到版本冲突阶段应写 state revision，遇到重复安装错误时应写 anti-rule。

**Goal Interruption / Task Resumption**。
 它和 Recent Constraint Tracking 很像，但不完全一样。Recent Constraint Tracking 更像“刚给你约束，你还能不能记住”；Task Resumption 更像“中途被打断、切到别的子任务、过了一段时间再回来，你还能不能恢复到正确子目标和进度状态”。这类 failure 在 agent 环境里非常常见，而且很适合体现 context memory 和 state memory 的分工。

**Staleness / Applicability Judgment**。
 这和 Source Conflict Resolution 也相关，但更偏“旧经验不是错，只是不再适用”。比如之前某个 procedure 在 A 项目有效，但换到 B 项目就不该直接迁移；或者某个 state 是历史真值，但当前环境已经变了。这个能力能很好支撑你文档里“防止使用错误经验”“提升针对性”的论点。

### 创新 1：提出“能力缺陷驱动”的压缩视角

现有方法多按任务、预算、相似度或结构类型决定如何压缩；我们首次将 agent 执行反馈映射到显式能力缺陷，并让不同缺陷触发不同压缩/整理操作。
 这条就是你的主创新。

### 创新 2：提出异构记忆写入，而非统一摘要

同一段轨迹不会统一压成一段 summary，而会根据归因结果被编译成 state update、strategy template、anti-pattern rule 或 conflict evidence graph。
 这条体现方法形式上的新意。

### 创新 3：提出负经验可回放机制

不是只记成功经验，也不是只把失败用于离线训练，而是在 test-time 维护可检索的 anti-memory，用于 veto、rerank 和 repair。





你们的方法不要再表述成“我们做一个更好的压缩器”，而应该明确成：**我们做的是一种能力驱动的记忆编译器**。它和现有工作的根本区别在于，现有方法大多还是把长历史当成同一种东西处理，只是在“压多少、什么时候压、用什么结构压”上做优化；你们的方法则先判断 **这段历史暴露的是哪一种能力问题**，再决定把它编译成哪一种记忆对象。也就是说，你们不是在做统一摘要，而是在做**按能力分流的异构记忆写入与回放**。这和你现在文档里的 motivation 是完全对齐的：什么时候保留原文、什么时候写状态、什么时候抽 procedure、什么时候写 anti-rule，不应该是 task-based，而应该是 capability-based。claw

具体做法上，我建议你们把一次长轨迹先切成若干 **episode**，每个 episode 都对应一个相对完整的 agent 事件，比如一次 arXiv 检索流程、一次环境配置与报错修复、一次规则更新、一次冲突证据判定。然后在 episode 结束后，不是立刻让模型写总结，而是先做一次**能力诊断**。这个诊断不是很抽象的“这段话重要吗”，而是一个很具体的分类问题：这段 episode 主要是在暴露哪种能力缺陷。你们现在的五类就够用了：最近约束没保住，就是 Context Retention；旧规则没被新规则覆盖，就是 State Revision；以前成功的流程没有迁移过来，就是 Procedural Reuse；以前犯过的错又犯了一次，就是 Error Avoidance；多个来源互相冲突但没选对，就是 Evidence Arbitration。这个分类可以先不用训练复杂模型，第一版直接用一个强模型做结构化判别就够了。输入给它的是：当前 episode 的轨迹、环境反馈、成败结果、以及和历史相似 episode 的对比；输出不是一句自然语言，而是一个结构化标签，比如 `type=state_revision, confidence=0.82, trigger=version_conflict`。这样你们后面所有写入动作就都有了明确入口。

关键在下一步：**不同标签对应不同写入模板**。这才是你们和别人的核心区别。比如如果分类结果是 Context Retention，那这段历史根本不应该被抽成高层摘要，因为问题不是“理解过去发生了什么”，而是“当前任务需要的近场信息不能丢”。这种情况下最好的写法是直接把最近原文保留下来，再抽几个关键槽位，比如路径、输出格式、截止时间、当前子目标。也就是说，这类 episode 被写进的是一个很轻量的 **Context Cache**，它里面存的不是总结，而是 `recent_raw_span + key_slots + unfinished_subgoal`。如果分类结果是 State Revision，就不要再写自由文本摘要，而要写成一个带版本链的状态对象，例如 `key=torch_version, old=2.1, new=2.3, source=pyproject.toml, supersedes=state_17`。这样你们的方法天然就支持“最新版覆盖旧版”，而不是像很多 summary-based 方法那样把新旧信息糊在一段文本里。对于 Procedural Reuse，也不该写“我们之前解决过类似问题”，而应该写成真正可复用的 procedure：触发条件是什么、步骤顺序是什么、在哪一步检查什么、什么情况下终止。比如前一次成功的 arXiv 文献检索就可以被编译成：`trigger=paper_search_task; steps=[query -> filter_year -> filter_topic -> dedup -> export_csv]`。对于 Error Avoidance，更不能只写“之前错了”。你们应该单独维护一个 **Anti-Memory**，里面存的是错误模式、诱发条件和禁止规则，比如 `trigger=environment_setup && has_cuda; veto=do_not_install_before_checking_cuda_compatibility; checklist=[nvidia-smi, nvcc --version, python version]`。这就是你们最有辨识度的地方，因为大多数方法即使会记失败，也只是把失败当经验的一部分，而不是把它变成未来可以直接 veto 当前动作的负记忆。最后，Evidence Arbitration 对应的不是普通 summary，而是一个小型 evidence object：记录有哪些来源、时间顺序如何、谁更可信、最终为什么采纳某一条。比如用户口头说 torch 2.1，但 `pyproject.toml` 和错误日志都指向 2.3，那么这里就写成一个 evidence record，而不是一句“后者更可信”。

所以从存储上看，你们的方法不是一个统一向量库，也不是一个统一 DAG summary，而是一个**五仓结构**。最底层永远保留一个 immutable raw log，原始消息和 tool output 不删，这是为了保证可追溯性；在它之上有 Context Cache，负责近场任务连续性；有 State Memory，负责存事实和版本链；有 Procedural Memory，负责存成功步骤模板；有 Anti-Memory，负责存负经验和禁止规则；有 Evidence Store 或轻量 Evidence Graph，负责存冲突证据。这里你们和 LCM / lossless-claw 的区别就很清楚了：LCM 的核心还是“把旧历史压成可以展开的 summary node”，它解决的是**怎么不丢历史**；你们的方法解决的是**即使不丢历史，也不能把所有历史都写成一种 summary**。你们是在 LCM 之后又往前走了一步：把“可追溯摘要”升级成“能力驱动的异构记忆对象”。claw

回放的时候同样不能再走“当前 query 做一次向量检索”这条老路。你们的方法应该先做一个很轻量的 **能力路由**。也就是说，当前任务一来，先判断它更像哪种问题：如果是最近约束型任务，比如“按照刚才的条件搜索 2024–2025 的 arXiv 文献并导出 CSV”，那就优先读 Context Cache，而不是去 procedural memory 里找一个高层模板；如果是规则更新型任务，比如“现在把环境从 Python 3.10 改成 3.11 且必须用 uv”，那就优先读 State Memory，看最新状态链，而不是把过去的原始对话一股脑塞进去；如果是相似流程迁移，比如“上次整理 arXiv 的流程现在迁移到 ACL/ICLR”，那就去 Procedural Memory 里拿 steps；如果是重复犯错风险高，比如又开始装 CUDA 环境，那就先查 Anti-Memory，看有没有 veto rule；如果是多源冲突，那就去 Evidence Store。也就是说，你们的方法不是“一个库统一检索”，而是**先决定该从哪种记忆里取什么，再回放给 agent**。这就是你们和 Mem0、A-MEM 这类统一 memory store 的核心差别：它们更偏“抽取—存储—检索”，而你们是“诊断—编译—定向回放”。claw

如果你问“那压缩到底发生在哪里”，答案是：压缩发生在**编译阶段**，不是发生在“把全文变短”的阶段。比如对 State Revision 来说，压缩就是把若干轮围绕一个配置版本的对话，压成一个带版本关系的 state object；对 Procedural Reuse 来说，压缩就是把一次完整尝试抽成 3–5 步模板；对 Error Avoidance 来说，压缩就是把一次失败与修复过程压成一个 anti-rule 和 checklist。也就是说，你们的压缩对象不是 token，而是 **episode-level decision structure**。这点很重要，因为这样你们就不会和 ACON 那种“学习压缩 guideline”混在一起。ACON 解决的是怎样在不掉太多性能的前提下把上下文压得更短；你们解决的是怎样把不同类型的历史压成不同语义的记忆对象。前者关注的是 compression ratio 和 task utility 的平衡，后者关注的是 **memory form 与 capability failure 的匹配**。claw

所以你们方法真正的创新点，最好不要写成泛泛的“ability-based compression”，而应该写成这样一句更硬的话：**我们提出了一种 capability-routed memory compiler，它将 agent 的长轨迹先诊断为显式能力缺陷，再分别编译为 context cache、versioned state、procedural template、anti-rule 和 evidence object，并在后续任务中按能力类型进行定向回放。** 这句话里已经把你们和现有工作的区别说清楚了：不是统一摘要，不是统一 memory store，不是统一经验提炼，而是**能力诊断驱动的异构记忆编译**。

如果把它再说得更直白一点，你们的方法和别人最大的区别就是三件事。第一，别人通常先压缩，再希望模型自己学会怎么用；你们是先判断“为什么会错”，再决定“该压成什么”。第二，别人往往只有正经验或统一经验库；你们把负经验单独建成 Anti-Memory，让它可以 veto 和 repair。第三，别人通常把 retrieval 做成“当前 query 找最相似历史”；你们做成“当前任务属于哪种能力需求，就去对应的 memory type 里取最合适的对象”。这三点一旦写清楚，创新边界就很稳了

## related work：

### benchmark侧

#### LoCoMo

LoCoMo 是 ACL 2024 的长期对话记忆 benchmark，特点是对话很长，平均每个对话约 300 turns、9K tokens、跨最多 35 个 session，还同时覆盖问答、事件摘要和多模态对话生成。它的重要价值是：**第一次比较系统地把“长期对话记忆”做成了标准测试集**。但它的局限也很明显：更偏**离线长对话理解**，不是 agent 环境，也不强调压缩策略本身，更不评估预算—性能曲线。

#### LongMemEval

LongMemEval 面向聊天助手的长期交互记忆，提出了五类能力：信息抽取、多 session 推理、时间推理、知识更新和 abstention。它很有价值，因为它把“更新”和“拒答”也纳入了长期记忆评测；论文还指出现有商业助手和长上下文模型在持续交互下会出现明显掉点，并提出了 session decomposition、fact-augmented key expansion、time-aware query expansion 等设计。局限是：它仍然主要评测**memory retrieval design**，而不是完整的**在线压缩决策**。

#### MemoryAgentBench

MemoryAgentBench 很接近你们要做的 benchmark 气质。它明确提出 memory agent 的四个核心能力：accurate retrieval、test-time learning、long-range understanding、selective forgetting，并把现有长文本数据改造成 incremental multi-turn 形式。它的贡献是把“多轮增量交互”放到了中心。局限在于：它更偏**memory 能力总评测**，还没有把 **compression fidelity / support traceability / budget sensitivity / task utility** 做成核心指标。

#### WildClawBench

WildClawBench 把 agent 扔进真实 OpenClaw 环境里做 60 个 end-to-end 任务，这是很重要的信号：**OpenClaw 已经变成一个真实 agent benchmark playground**。但 WildClawBench 的目标是“整体 agent 能力”，不是专门研究 context compression / memory management；它可以成为你们 benchmark 的环境底座或外部验证集，但本身不是压缩 benchmark。

#### SWE-ContextBench

SWE-ContextBench 不是对话 benchmark，但非常值得看。它专门评估 agent 如何跨相关任务复用先前经验，并同时看 accuracy、time efficiency、cost efficiency。更关键的是，它发现**正确选择的 summarized context** 能提升准确率并显著降低成本，而错误或无过滤的 context 可能有负作用。这个发现非常适合作为你们方法 paper 的理论动机。

#### ACBench

ACBench 研究的是**模型压缩**对 agent 能力的影响，不是 prompt/context 压缩。但它有两个启发：第一，agent 场景确实需要单独 benchmark；第二，评估要覆盖 workflow、tool use、long-context understanding、real-world application 等多个维度，而不能只看 QA accuracy。你们做 context compression benchmark 时可以继承这种“能力矩阵”思路

### 方法侧

#### ACON

ACON 是很强的直接相关工作。它把问题定义成长时程 agent 的上下文压缩，并且明确报告了 accuracy–peak tokens 的 trade-off：在 AppWorld / OfficeBench / Multi-objective QA 上可减少 26–54% peak tokens，同时尽量保持性能；其核心是**学习压缩 guideline**，不是手工规则。

#### MemGPT

MemGPT 的核心思想是把 memory 做成类似操作系统的分层存储，管理不同 memory tiers，在多 session chat 和大文档分析中扩展有效上下文。它是“分层记忆架构”这条线的代表。弱点在于：它更多是**memory paging / retrieval** 范式，未必直接优化“压缩后信息是否可追溯、可恢复”。

#### MemoryBank

MemoryBank 更早，强调长期用户画像与记忆更新，还借鉴遗忘曲线来做记忆强化与遗忘。这个方向说明“selective forgetting”是合理议题，但它更偏个性化陪伴，不够 agentic，也不够强调在线预算。

#### Mem0

Mem0 是这条线里很重要的一篇。它做动态提取、整合、检索 salient information，还进一步做 graph memory；在 LoCoMo 上报告了比若干 baseline 更好的整体效果，同时大幅降低延迟和 token cost。它代表的是**structured persistent memory** 路线。弱点是：它更像“抽取—存储—检索”的 memory system，还不是把**压缩决策**显式建模成一个预算控制问题。

#### A-MEM

A-MEM 把 Zettelkasten 式结构化组织和 agent-driven memory decision 结合起来，更强调记忆网络的动态重组和 agentic control。它说明：**memory 不是只靠向量检索，也可以有结构化组织**。这一点和你们想做“整理/总结/压缩”非常一致。

#### HiAgent

HiAgent 专注 working memory，而不是跨 session 长期记忆。它的关键 insight 是：用 **subgoal** 作为 memory chunk 单位，让 agent 主动把之前子目标的 action-observation pairs 压成摘要，只保留当前子目标相关的细节。对你们来说，这是很强的 baseline，也是一条可借鉴的设计思路：**摘要单元最好不是固定窗口，而是任务结构单元。**

#### Focus / Active Context Compression

Focus 提出 agent 自主决定何时 consolidate 到 persistent knowledge block、何时 prune raw history，在 SWE-bench Lite 上报告了不掉准确率的 token reduction。虽然规模还不大，但它很重要，因为它把“压缩时机决策”从被动触发变成了**agent-controlled action**。

#### ContextBudget / BACM-RL

ContextBudget 更进一步，直接把 context management 形式化为**受 token budget 约束的序列决策问题**，并用 RL 学习在不同预算下何时压缩、压多少。这条线非常值得你们吸收，因为顶会会更喜欢“压缩不是 heuristic，而是可学习的决策策略”。

#### OpenClaw lossless-claw / LCM

这是你提到的重点。lossless-claw 是 OpenClaw 的 plugin，核心是用 **DAG-based summarization** 替换内建 sliding-window compaction，把每条消息保存在 SQLite 中，再把旧消息压成 summary nodes。LCM 论文则进一步明确提出 dual-state memory（Immutable Store + Active Context）、hierarchical summary DAG、lossless pointers、lcm_expand / lcm_grep、软硬阈值 compaction、zero-cost continuity 等设计，并宣称在 OOLONG 长上下文评测上优于 Claude Code。这个方向最大的贡献不是“摘要更好”，而是**把 summary 变成可展开、可追溯、可回到原文的中间层**。这对你们的研究至关重要。

#### OpenClaw Context Optimizer
